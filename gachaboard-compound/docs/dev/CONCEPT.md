# コンセプト・設計思想

Gachaboard の設計の根底にある考え方。

---

## 1. プロジェクトコンセプト

### 1.1 何を作っているか

| 項目 | 内容 |
|------|------|
| **製品** | 音楽・映像・デザインファイルを**貼り付けて共有**できるリアルタイム共同ホワイトボード |
| **対象ユーザー** | Discord コミュニティ内の身内。URL を知っていれば参加可能 |
| **運用方針** | ローカルサーバー 1 台で完結。クラウド依存を最小化 |
| **管理単位** | ワークスペース（3〜4 プロジェクト想定）単位でボードを管理 |
| **ライセンス** | Apache 2.0（compound / tldraw ベース） |

### 1.2 想定ユースケース

- **クリエイター同士のレビュー**: 動画・音声を貼ってコメントし合う
- **プロジェクト共有**: デザイン案・コード・素材をボードに集約
- **ブレスト**: 画像・リンク・メモを並べて議論
- **身内だけの閉じた空間**: Discord 認証で匿名を排除

---

## 2. 設計原則

### 2.1 依存の最小化

- **クラウド SaaS にしない**: PostgreSQL、MinIO、自前 sync-server で完結
- **Tailscale でネットワーク構築**: グローバル IP・ポート開放不要。最大 100 台（無料）
- **S3 互換**: AWS S3 でも MinIO でも動く

### 2.2 スケールの前提

| 項目 | 想定値 | 備考 |
|------|--------|------|
| 同時接続 | 最大 30 人 | y-websocket-server 単体で十分 |
| 1 ボードあたりシェイプ | 約 1000 | パフォーマンス検証の目安 |
| 100 人超 | 別構成 | Hocuspocus 等の検討 |

### 2.3 認証方針

- **Discord OAuth 必須**: 匿名排除。身内運用を前提
- **ワークスペース境界**: メタデータ・ボード・アセットは workspace で分離

## 3. compound の位置づけ

### 3.1 なぜ compound か

| 観点 | 内容 |
|------|------|
| **ライセンス** | tldraw の Apache 2.0 フォーク。商用利用しやすい |
| **共同編集** | Yjs + CRDT によるリアルタイム同期を前提とした設計 |
| **カスタムシェイプ** | メディア・ファイル特化のシェイプを自由に拡張可能 |

### 3.2 fresh からの移行

- **compound が本番候補**: fresh（tldraw）から機能を移植し、compound 上で本番運用
- **認証・ワークスペース・API 等**: 共通設計で fresh と互換
- 詳細は [archive/fresh-to-compound-migration.md](../archive/fresh-to-compound-migration.md)

## 4. データの流れ

```
[ブラウザ A]                    [ホスト 1 台]
       │                              │
       ├── Tailscale P2P ──────┐      ├── nextjs-web :3000
       │                       │      ├── sync-server :5858
[ブラウザ B]                   └─────►├── postgres :5432
       │                              └── MinIO / uploads :9000
       └── WebSocket ─────────────────► Y.Doc / Awareness 同期
```

- **Tailscale**: 端末間 P2P 暗号化トンネル
- **Y.Doc**: CRDT によるドキュメント同期。オフライン編集もマージ可能
- **Awareness**: カーソル・ユーザー情報（永続化しない）

---

## 5. ファイル保存の考え方

- **S3/MinIO 必須**: ファイルは S3 互換ストレージにのみ保存
- 開発・検証: `docker compose up -d` で MinIO を起動
- 本番: MinIO または AWS S3 で永続化

---

## 6. メディア変換の方針

| 入力 | 変換 | 目的 |
|------|------|------|
| 動画（mp4 等） | 720p H.264 + サムネイル | 再生負荷軽減 |
| WAV | MP3 | ブラウザ互換・容量削減 |
| URL | OGP プレビュー | リンクの可視化 |

- **fluent-ffmpeg**: 変換処理。ffmpeg バイナリ必須
- 変換は非同期。完了まで「変換中...」表示
