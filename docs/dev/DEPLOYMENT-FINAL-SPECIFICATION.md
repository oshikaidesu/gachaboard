# 配布・運用の最終構成

Gachaboard の**配布形態**と**ホスト運用**の最終方針をまとめています。
検討の結果、**軽量 VM 内でサーバーを動かす**構成を採用しました。

---

## 1. 方針サマリー

| 項目 | 決定 |
|------|------|
| **配布形態** | 軽量 Linux VM 内で Docker（Postgres, MinIO, sync-server）+ Next.js を稼働 |
| **クロスプラットフォーム** | Windows / macOS / Linux で同じ VM イメージ・手順を使用 |
| **Electron** | 採用しない（バイナリ同梱が重く、複雑になるため） |
| **sync-server** | 現状どおり別プロセス。Next.js への統合は行わない |
| **Docker** | VM 内で使用。ホスト側の Docker は不要（VM 内で完結） |

---

---

## 3. メリット

| メリット | 内容 |
|----------|------|
| **ホストと分離** | AE や開発環境とリソース・ポートを分離。競合しない |
| **クロスプラットフォーム** | VM 内は同じ Linux。ホスト OS に依存しない |
| **管理が単純** | 配布は「VM イメージ or セットアップ手順」1 パターン |
| **軽量** | 2〜4GB 程度のメモリで稼働可能 |
| **運用が単純** | VM 起動 = Gachaboard 稼働、VM 停止 = 停止 |

---

## 4. ポート構成

| ポート | サービス | ホストからのアクセス |
|--------|----------|----------------------|
| 3000 | Next.js | ホスト:3000 → VM:3000 に転送 |
| 9000 | MinIO（Presigned URL 用） | ホスト:9000 → VM:9000 に転送 |
| 5858 | sync-server | VM 内のみ（Next.js の rewrite で /ws 経由） |
| 5432 | PostgreSQL | VM 内のみ |

参加者がアクセスするのは **3000** と **9000**。Tailscale 運用時はホストの Tailscale アドレス経由。

> **注**: 上記は VM 内の構成例。通常の docker-compose（リポジトリ付属）では、ホストからは Next.js 18580・sync-server 18582・PostgreSQL 18581・MinIO 18583 にマッピングされます。詳細は [ENV-REFERENCE.md](../user/ENV-REFERENCE.md) を参照。

## 5. VM の推奨スペック

| 項目 | 推奨値 |
|------|--------|
| OS | Alpine Linux または Ubuntu Server（ミニマル） |
| メモリ | 2GB（最小）〜 4GB（余裕） |
| CPU | 2 vCPU |
| ディスク | 20GB 以上（ボリューム用に余裕を持たせる） |
| GUI | 不要（ヘッドレス） |

## 6. クロスプラットフォーム対応

| ホスト OS | VM ツール | 備考 |
|-----------|-----------|------|
| Windows | Hyper-V, VirtualBox, UTM, Multipass | Multipass は軽量で扱いやすい |
| macOS | UTM, Parallels, Multipass, Lima | Apple Silicon は UTM など対応ツールを選択 |
| Linux | KVM, VirtualBox, Multipass | Multipass で統一可能 |

**VM 内環境は共通**: Alpine または Ubuntu Server + Docker + docker-compose

---

## 7. 配布物

| パターン | 内容 |
|----------|------|
| **A: セットアップ手順** | VM 作成 → Linux インストール → Docker インストール → Gachaboard クローン → docker compose up の手順書 |
| **B: プリビルド VM イメージ** | Gachaboard が入った VM イメージを配布（OVA 等）。起動してポート転送設定のみ |
| **C: Multipass スクリプト** | `multipass launch` 用の cloud-init やスクリプトでワンコマンド起動 |

---

## 8. 採用しなかった案と理由

| 案 | 理由 |
|----|------|
| **Electron 同梱** | Postgres, MinIO バイナリ同梱が重く、複雑。パッケージ 300MB 超 |
| **sync-server を Next.js に統合** | カスタムサーバーが必要で Next.js の利点が薄れる |
| **SQLite + ローカルストレージ** | Presigned URL を捨てる設計になり、大容量転送時の負荷が増える |
| **リバースプロキシでポート統一** | ホスト名の管理や設定が増え、運用が複雑になる |
| **別物理マシン必須** | 必須ではなく、VM で同じマシン内に分離できる |

## 9. Discord 認証

VM 構成でも変わらない。

- **NEXTAUTH_URL**: 参加者がアクセスする URL（例: `http://host-tailscale:3000`）
- **Discord Redirect URI**: 上記に合わせて登録
- ポートは 3000 のまま。VM のポート転送で透過的にアクセス可能
