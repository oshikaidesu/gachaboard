# 24時間運用時の注意点

Gachaboard を常時稼働させる際に気をつけるべき点をまとめました。

---

## 1. プロセス管理（Next.js の自動再起動）

`scripts/start/production.sh` は `npm run start` をバックグラウンドで起動するだけなので、**クラッシュや OOM で落ちても自動では再起動しません**。ターミナルを閉じるとそのままプロセスも終わります。

**対策**（手順は [AUTO-RESTART.md](AUTO-RESTART.md) を参照）:

- **systemd（Linux）** または **PM2（Mac/Linux）** で Next.js をサービス化すると、落ちたときに自動で再起動されます。サンプルは `scripts/systemd/gachaboard-web.service.example`。

### Windows での常時起動（Servy）

Windows でコマンドプロンプトを閉じても落ちないようにするには、**Servy** で Gachaboard を Windows サービスとして登録する方法があります。GUI（デスクトップアプリ・Manager）で起動・停止・ログ確認ができます。

**Servy の設定例**（Tailscale 本番で 24 時間運用する場合）:

- **プログラム**: `powershell.exe`
- **引数**: `-NoProfile -ExecutionPolicy Bypass -File "d:\server\gachaboard\scripts\win\run.ps1" -Tailscale`
- **作業ディレクトリ**: `d:\server\gachaboard`（プロジェクトのルートを実際のパスに合わせる）

Localhost 本番のみの場合は、引数から `-Tailscale` を外して `-File "…\scripts\win\run.ps1"` だけにします。

**運用の流れ**:

- 起動・停止・再起動: Servy の **Manager アプリ** で操作する。
- ログ: Manager のログビューア、または Servy が出力するログファイルを参照する。
- コードや `.env` の更新後: 一度サービスを停止 → 必要な作業（`npm install` やビルドなど）→ 再度 Servy で開始する。

起動タイプを「自動」にすると、PC 再起動後もサービスが自動で立ち上がります。

---

## 2. sync-server のメモリ・ディスク

sync-server は Hocuspocus + SQLite（`YPERSISTENCE`）でデータを永続化しています。再起動しても Y.Doc は SQLite から復元されます。

気をつけたいこと:

- **メモリ**: 同時に開いているボードが増えるほどメモリを使います。長時間運用でじわじわ増えていないか、たまに確認しておくと安心です。
- **ディスク**: `data/sync` の SQLite ファイルがボード数・更新頻度に応じて大きくなります。

**対策**:

- メモリ使用量を定期的に確認する。
- `data/sync` の容量を監視し、必要に応じてアラートを設定する。

---

## 3. ディスク枯渇・バックアップ

- **ディスク容量**: 定期的な不要ファイルの削除や容量監視は現状未整備です。
- **PostgreSQL**: 定期バックアップの手順はドキュメント化されていません。`pg_dump` を使って定期的にバックアップしておくことを推奨します。
- **MinIO**: 現状は単一障害点です。バックアップ・冗長化は別途検討してください（[database-and-storage-inventory.md](../dev/database-and-storage-inventory.md) 参照）。

---

## 4. ログのサイズ

systemd や PM2 でログローテーションを設定しないと、長期運用でディスクを圧迫することがあります。

- systemd: `journald.conf` で `SystemMaxUse` を設定する。
- PM2: `pm2 install pm2-logrotate` でローテーションを有効にする。

---

## 5. レートリミット（in-memory）

招待 API・OGP API のレートリミットはメモリ上で管理しています。Next.js を再起動するとリセットされ、複数インスタンスではインスタンスごとにカウントが別になります。

単一インスタンスの常時運用であれば大きな問題にはなりにくいですが、再起動直後は制限が緩くなります。より厳密にしたい場合は、リバースプロキシ（nginx / Caddy）や Redis ベースのレートリミットを検討してください（[SECURITY.md](../../SECURITY.md) 参照）。

---

## 6. DB スキーマ変更

現状、本番でも `prisma db push` を使っています。マイグレーション履歴がないため、スキーマの不整合が起きたときにロールバックが難しくなります。本番運用では `prisma migrate` への移行を検討してください。

---

## 7. Tailscale で 24 時間運用する場合

- ホスト PC がスリープ・シャットダウンするとサービスが止まります。
- ノート PC の場合はスリープを無効にしておく必要があります。
- 停電対策として **UPS** の導入も検討してください。

---

## 8. ヘルスチェック

sync-server は `/health` エンドポイントを持っています。Next.js は `/api/health` でヘルスチェックに応答します。監視ツールやロードバランサと組み合わせる場合はこれらを使ってください。

---

## 9. 長時間開いていると「消える・切れる」ことがあるもの

サーバー側のデータは SQLite に永続化されているため、長時間起動していてもボードの内容が勝手に消えることはありません。

ブラウザ側では、以下の有効期限によって「表示が消えた」「接続が切れた」ように見えることがあります。

| 対象 | 有効期限 | 挙動 |
|------|----------|------|
| **画像・動画・ファイルの表示 URL** | 1時間（Presigned URL） | 1 時間以上開いたままにするとアセットの読み込みが失敗し、表示が消えたように見える。データ自体は消えておらず、ページを再読み込みすれば再表示される。 |
| **ログインセッション** | 30日（NextAuth デフォルト） | 30 日間再ログインしないとセッションが切れてログアウトされる。 |
| **sync 接続用トークン** | 30分（24 分ごとに自動更新） | タブを長時間バックグラウンドに置いておくとトークン更新が遅れ、WebSocket が切断されることがある。再読み込みで再接続される。 |

画像・動画が「消えた」ように見えたら、まずページを再読み込み（F5 など）してみてください。

---

## 10. 長時間運用で DB 接続が不安定になる場合

長時間稼働していると、PostgreSQL や MinIO への接続がたまに失敗することがあります。主な原因は、サーバー側や NAT・ファイアウォールでアイドル接続が切られたあと、アプリ側の接続プールがその「死んだ」接続を返してしまうことです。

**本番での対策**:

- **PostgreSQL**: `nextjs-web/src/lib/db/db.ts` で接続プールに `connectionTimeoutMillis` と `idleTimeoutMillis` を設定しています。アイドル接続を一定時間で明示的に廃棄し、サーバー側で切られる前に新しい接続を張り直します。
- **MinIO/S3**: 接続エラー時にクライアントを破棄し、次回リクエストで新しい接続を作るようにしています。
- **スタートアップスクリプトで再起動**: Mac/Linux の launcher や Windows の run.ps1 は「起動 = 再起動」の意味を持たせています。同じマシンでもう一度スタートアップを実行すると、既存の Next.js を止めてから起動し直すため、接続がリセットされて解消することがあります。

### 再起動時のデータとユーザー操作

**再起動してもデータは消えません。** 止めるのは Next.js プロセスだけです。PostgreSQL・MinIO・sync-server は別プロセスで動いており、データはディスクに永続しています。

**ユーザーはページをリロード（F5）すれば十分です。** 再起動中の数秒〜数十秒は「接続できない」などになることがありますが、リロードすれば新しい Next.js に繋がり直し、同じワークスペース・ボードが表示されます。編集中の内容も sync-server が動いたままであれば、リロード後に WebSocket で再接続して同期が続きます。

---

## まとめ

| 優先度 | 内容 | 対応方法 |
|--------|------|----------|
| 高 | Next.js が落ちたら自動復旧しない | systemd / PM2（Mac/Linux）、**Servy**（Windows）でサービス化する |
| 高 | ディスク枯渇で DB・MinIO・sync が止まる | 容量監視・アラート、バックアップ方針を決める |
| 中 | ログでディスクが埋まる | logrotate / PM2 ログローテーションを設定する |
| 中 | 本番の DB 変更が危険 | `prisma migrate` への移行を検討する |
| 中 | 長時間運用で DB/sync がたまに失敗する | Prisma プール設定（db.ts）、起動スクリプトで再起動 |
| 低 | レートリミットが再起動でリセット | 必要なら Redis 等を検討する |
| 環境依存 | ホストのスリープ・停電 | スリープ無効化・UPS（Tailscale 運用時） |
