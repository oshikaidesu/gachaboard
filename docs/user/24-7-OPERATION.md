# 24時間運用時の注意点

Gachaboard を**常時稼働（24時間運用）**させる際の懸念点と対策のまとめです。

---

## 1. プロセス管理（Next.js の永続化）

**懸念**: `scripts/start/production.sh` は `npm run start` をバックグラウンドで起動するだけで、**プロセス監視・自動再起動は行いません**。クラッシュや OOM で落ちるとそのまま停止し、ターミナルを閉じるとプロセスも終了します。

**対策**（手順は [AUTO-RESTART.md](AUTO-RESTART.md) を参照）:

- **Docker で一式動かす**: `docker compose --profile app up -d` で起動すると、`web` コンテナに `restart: unless-stopped` が設定されているため、落ちても自動で再起動されます。
- **Docker を使わない場合**: systemd（Linux）または PM2（Mac/Linux）で Next.js をサービス化し、クラッシュ時に自動再起動する。サンプルは `scripts/systemd/gachaboard-web.service.example`。

---

## 2. sync-server のメモリ・ディスク

**現状**: sync-server は Hocuspocus + **SQLite**（`YPERSISTENCE`）で永続化しています。再起動しても Y.Doc は SQLite から復元されます。未使用ルームはメモリ解放の恩恵がありますが、同時接続が多いとメモリ使用量が増えます。

**懸念**:

- **メモリ**: 同時に開かれるボード数に応じてメモリが増加。長時間運用でリークや増え続けがないか要観察。
- **ディスク**: `compound_sync_data` ボリューム内の SQLite が、ボード数・更新頻度に応じて肥大化する可能性。

**対策**:

- メモリ: Docker の `mem_limit` を設定し、OOM 時はコンテナ再起動に任せる。
- ディスク: `compound_sync_data` の容量を監視し、必要に応じてアラートを設定する。

---

## 3. ディスク枯渇・バックアップ

**懸念**:

- **ディスク容量枯渇**: 定期的な不要ファイル削除・容量監視は未整備です。
- **PostgreSQL**: バックアップ手順はドキュメント化されていません。
- **MinIO**: 単一障害点。バックアップ・冗長化は別途検討が必要です（[database-and-storage-inventory.md](../dev/database-and-storage-inventory.md) 参照）。

**対策**:

- ディスク使用量の監視（アラート）を設定する。
- PostgreSQL の定期バックアップ（`pg_dump` 等）を運用で用意する。
- MinIO（S3）のバックアップまたはライフサイクルポリシー（古いオブジェクト削除など）を検討する。

---

## 4. ログの増大

**懸念**: Docker の場合はコンテナの stdout がログになるが、**ローテーションの上限を設定していないとディスクを圧迫**する可能性があります。systemd や PM2 で動かす場合も同様です。

**対策**:

- `docker-compose.yml` の `logging` で `max-size` / `max-file` を設定する。
- 非 Docker の場合は logrotate 等でログローテーションを設定する。

---

## 5. レートリミット（in-memory）

**現状**: 招待 API・OGP API のレートリミットは **in-memory** 実装です。Next.js を再起動するとリセットされ、複数インスタンスではインスタンスごとに別カウントになります。

**懸念**: 24時間単一インスタンス運用では大きな問題にはなりにくいが、再起動直後は制限が緩む。

**対策**: より厳格な制限や複数インスタンス対応が必要な場合は、リバースプロキシ（nginx / Caddy）や Redis ベースのレートリミットを検討してください（[SECURITY.md](../../SECURITY.md) 参照）。

---

## 6. DB スキーマ変更（db push）

**現状**: 本番でも `prisma db push` を使う前提の記述があります。マイグレーション履歴がないため、本番でスキーマ不整合が起きた場合のロールバックが難しくなります。

**対策**: 本番運用では `prisma migrate` への移行を検討してください（[yjs-system-specification.md](../dev/yjs-system-specification.md) 参照）。

---

## 7. Tailscale で 24 時間運用する場合

**懸念**（[yjs-system-specification.md](../dev/yjs-system-specification.md) より）:

- ホスト PC がスリープ・シャットダウンするとサービス停止。
- ノート PC などではスリープ設定の見直しや**スリープ無効化**が必要。
- 停電に備えて **UPS** の検討が望ましい。

---

## 8. 監視・ヘルスチェック

**現状**: sync-server には `/health` があり、Docker の healthcheck で利用されています。Next.js アプリ用の HTTP ヘルスチェックエンドポイントは用意されていません。

**対策**: オートヒーリングやロードバランサで監視する場合は、必要に応じて `/api/health` 等のエンドポイントを用意してください。

---

## 9. 長時間起動で「勝手に消える」・切れるもの

**サーバー側**: sync-server の Y.Doc は SQLite に永続化されているため、**長時間起動してもサーバー側でボード内容が勝手に消えることはありません**。

**クライアント（ブラウザ）側**で、有効期限により「表示が消えた」「つながらない」ように見えることがあります。

| 対象 | 有効期限 | 挙動 |
|------|----------|------|
| **画像・動画・ファイルの表示 URL** | **1 時間**（Presigned URL） | ボードを 1 時間以上開いたままにすると、アセットの読み込みが 403 等で失敗し、表示が消えたように見える。**データは消えておらず、ページを再読み込みすると再度 URL が発行されて表示される。** |
| **ログインセッション** | **30 日**（NextAuth デフォルト） | 30 日間再ログインしないとセッションが期限切れになり、ログアウトされる。 |
| **sync 接続用トークン** | **30 分**（24 分ごとに再取得） | タブがバックグラウンドで長時間止まっていると、トークン更新が遅れ、WebSocket が切断されることがある。**再読み込みで再接続される。** |

**対策の目安**:

- 画像・動画が「消えた」ように見えたら → **ページを再読み込み**（F5 など）。
- 長時間開いたタブでは、Presigned URL の有効期限（1 時間）を超えるとアセットだけ読み込めなくなる仕様のため、必要なら「一定時間で再取得」する実装の検討余地あり（現状は未実装）。

---

## まとめ（優先度の目安）

| 優先度 | 懸念 | 対策の方向性 |
|--------|------|----------------|
| 高 | Next.js が落ちたら復旧しない | Docker `web` で動かすか、systemd/PM2 で自動再起動 |
| 高 | ディスク枯渇で DB・MinIO・sync が止まる | 容量監視・アラート、バックアップ方針の策定 |
| 中 | ログでディスクが埋まる | Docker logging の max-size 設定 / logrotate |
| 中 | 本番の DB 変更が危険 | `prisma migrate` への移行を検討 |
| 低 | レートリミットが再起動でリセット | 必要なら Redis 等を検討 |
| 環境依存 | ホストのスリープ・停電 | スリープ無効化・UPS（Tailscale 運用時） |
